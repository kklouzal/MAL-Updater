# Current Status

## Repository / packaging state

- The repo is treated as the canonical skill package (`SKILL.md` at repo root).
- Runtime state is externalized to `.MAL-Updater/` by default instead of living under the repo tree.
- The bootstrap/onboarding surface starts with `bootstrap-audit`, which now exposes provider readiness, per-provider operation-mode guidance/next-command hints, runtime-initialization readiness, user-daemon install/drift readiness, explicit manual-vs-daemon operation expectations, provider-intent/partial-bootstrap counts for partial-install edge cases, secrets-dir permission posture, blocking vs non-blocking onboarding counts, and explicit recommended commands for automation-friendly consumers.
- Repo-owned systemd automation is committed as a portable `mal-updater.service` template and rendered at install time by `scripts/install_user_systemd_units.sh`.
- The unattended model is now user-level systemd **daemon-first**, with internal Python control loops rather than timer-driven one-shot jobs.
- Public-repo hygiene is now part of the project contract: tracked code/references/tests/history should remain anonymized and free of personal identifiers, host-local absolute paths, and real secrets.

## Working today

- Python worker/application exists
- SQLite bootstrap + migrations exist
- Crunchyroll auth bootstrap and live snapshot fetch exist
- MAL OAuth and guarded MAL apply exist
- mapping review / queue triage workflows exist
- exact-title split-bundle auto-resolution now covers conservative same-title TV suffix companions (for example `Title` + `Title (2009)`) when provider episode evidence cleanly fits the combined entry count and no stronger non-bundle rival remains
- supplemental-rescued exact-title overflow cases now also surface conservative bundle companions for review when the follow-up installment is only reachable through supplemental candidate IDs (for example `Girls Bravo`), while still preserving the rule that supplemental rescue inputs do not auto-resolve the mapping by themselves
- alias-driven exact-title overflow cases can now also surface conservative bundle companions for review when the franchise alignment only becomes obvious through MAL alternative titles, while still keeping exact bundle auto-resolution on the stricter primary-title-family rule
- when exact-overflow bundle review has multiple plausible numbered follow-up installments, companion selection now prefers the nearest contiguous follow-up season/part sequence over skip-ahead franchise noise with the same episode-fit envelope
- standalone installment-only provider season labels such as bare `III` or `2` now prefix the base title during MAL search generation and count as real season cues for scoring/auto-approval even when provider metadata omitted `season_number`
- explicit later-season searches can now also try a small franchise-specific sequel-alias table (for example `A Certain Scientific Railgun Season 2` -> `A Certain Scientific Railgun S`) and treat an exact alias-hit as explainable season evidence instead of leaving the sequel looking like a base-title false positive
- those generic season-query and franchise-specific sequel-alias paths now also reuse season numbers inferred from provider season titles when provider metadata omits `season_number`, so later-season lookups do not quietly fall back to weaker base-title-only search behavior on sparse provider metadata
- exact later-season classification now also honors those inferred provider season numbers, so sparse provider metadata does not leave otherwise clean explicit sequel resolutions stuck in review-only posture just because raw `season_number` metadata was absent
- the same small franchise-specific sequel-alias table now also contributes later-season context when the provider season title itself is already alias-labeled (for example `Demon Lord, Retry! R`), so exact alias-title hits and alias-labeled related candidates can still carry explainable season evidence even when provider metadata omitted `season_number`
- alias-query later-season evidence is now only granted when the candidate itself actually matches that franchise-specific sequel alias, so MAL base-title false positives returned under alias searches no longer inherit fake sequel confidence just because the query was alias-labeled
- relation expansion now also kicks in for explicit later-season base-series false positives, so if MAL only exposes the real sequel through a related-anime hop the mapper can still recover that sequel conservatively instead of getting stranded on the misleading base-series hit
- whole-season later-season provider titles now penalize split-part/cour candidates when the provider's observed episode evidence already fits a non-segmented season entry, so plain `Season 2` / `II` labels no longer tie with `Part 2` candidates unless the provider data itself looks like an aggregated split-cour shell
- long-lived daemon runtime + service manager exist
- service-status now exposes recent daemon loop/task state, API-usage snapshots, health snapshot parsing, and log-tail context for unattended debugging, plus a terse `--format summary` operator view for quick checks
- service task state now persists per-task cadence metadata (`every_seconds`), decision timing, last-run start/finish/duration, next-due timing, budget provider labels, budget backoff level (`warn` vs `critical`), adaptive failure backoff state, and active cooldown countdowns so unattended operators can see when a lane should run next vs when it is intentionally cooling down
- request-event logging / budget awareness scaffolding exists
- daemon budget skips now compute both warn-threshold pacing windows and hard critical recovery windows, persist per-task budget backoff state, and avoid re-check/log spam until the provider budget has room again
- service budget defaults are now provider-generic: non-MAL source lanes can inherit shared source-provider hourly/backoff/auth-failure defaults without being silently treated as Crunchyroll unless an explicit per-provider override exists
- daemon budget policy can now also be overridden per task/lane (`service.task_*` tables), with service state/status surfacing whether a cooldown came from task-level vs provider-level policy so MAL lanes like `sync_apply` no longer have to share one blunt provider-only budget posture
- the repo now ships built-in `sync_apply` task defaults for MAL hourly limit, projected request cost, learned-history depth, conservative learned p90 percentile, and warn/critical/auth-failure cooldown floors so fresh unattended installs inherit a conservative aggregate-apply posture without having to copy those numbers from example config first
- unattended `sync_apply` now also has an explicit bounded execution posture via `service.task_execute_limits` (default batch size `8`) instead of assuming a full aggregate pass by default
- when a lane's execution posture changes materially, the daemon now resets stale projected-request/backoff state for that lane so old pathological request history does not keep poisoning future unattended decisions
- unattended `mal_refresh` now also ships a small explicit cold-start projected request cost plus a shallow learned-history window, so fresh installs no longer treat MAL token refresh as an unknown zero-cost lane before the daemon has seen one real run
- daemon budget gating is now modestly projection-aware: budgeted lanes persist observed request deltas, can optionally take explicit `service.task_projected_request_counts` overrides or fetch-mode-specific `service.task_projected_request_counts_by_mode` overrides, and will pre-emptively warn/skip when the projected post-run hourly ratio would cross warn/critical thresholds instead of only reacting to the current raw count
- observed request-delta projections now keep a short rolling baseline overall and per fetch mode, smoothing incremental/full-refresh fetch budgeting so one unusually expensive run is less likely to dominate the next unattended budget decision
- learned request projections are now tunable per task and per provider: operators can shrink/expand the observed-history window and optionally switch a lane/provider from mean-style smoothing to a conservative percentile baseline when a fetch path is burstier than average; task-specific tuning still wins, and when no explicit task/provider percentile is configured the daemon auto-switches obviously bursty lanes onto a conservative learned p90 baseline instead of blindly trusting the smoothed mean
- the repo now also ships opinionated provider defaults for the current source providers instead of leaving all pacing policy as operator-only examples: Crunchyroll now defaults to a deeper learned-history window plus conservative percentile budgeting/backoff floors, while HIDIVE now also ships a conservative learned p90 percentile baseline alongside its quieter hourly/backoff/auth-failure defaults
- provider-task failures now trigger adaptive failure-aware cooldowns with persisted reason / failure class / configured floor / retry countdown / consecutive-failure state so auth-fragile fetch lanes do not thrash every loop after a bad run
- auth-style failure detection is now shared between daemon/runtime and health-check surfaces, so the same residue classes drive both failure cooldown classification and later operator rebootstrap guidance instead of those heuristics drifting apart
- bootstrap-audit and health-check now also surface a compact auth-failure classification (`invalid_grant`/revoked token, missing refresh material, malformed token payload, session-auth residue, or generic auth/login failure) alongside the raw reason so operators can see *why* reauth/rebootstrap is being recommended instead of only getting a generic auth-degraded label
- bootstrap-audit onboarding / `recommended_commands` output now also preserves stable machine-readable command metadata (`reason_code`, `automation_safe`, `requires_auth_interaction`) and carries `auth_failure_kind` / `auth_remediation_kind` on auth-driven reauth/rebootstrap steps; terse `bootstrap-audit --summary` now surfaces those remediation-classification fields plus the top recommended command and its reason-code / automation-safe / auth-interaction posture so bootstrap-degraded state can be triaged without dropping into the full JSON payload
- those auth-degradation payloads now also carry residue-specific remediation metadata (`auth_remediation_kind` / `auth_remediation_detail`), and deduped maintenance commands preserve the more specific auth reason-code when the same login/rebootstrap command also repairs a generic missing-state condition
- health-check now inspects persisted daemon failure state and can recommend provider auth re-bootstrap when repeated unattended fetch failures look auth-related (not just repeated 401/unauthorized loops, but also refresh/login failure residue, missing-refresh-token / missing-refresh-material residue, malformed-token-payload residue, and provider session-state `auth_failed` phases)
- the same health surface now also inspects the daemon's `mal_refresh` lane, so repeated unattended MAL token-refresh auth failures (for example `invalid_grant` or missing refresh material residue) surface an explicit `mal-auth-login` recommendation instead of reading like generic daemon churn
- `health-check-cycle` now accepts the same review/maintenance tuning knobs as `health-check` (`--review-issue-type`, `--review-worklist-limit`, `--mapping-coverage-threshold`, `--maintenance-review-limit`) so unattended/manual cycle runs no longer fall back to stale hard-coded triage defaults when newer backlog/coverage policy is being tuned
- provider fetch lanes now persist successful fetch-mode state plus a periodic full-refresh anchor so unattended daemon runs can stay incremental by default while still forcing a conservative provider `--full-refresh` sweep on a configurable cadence (`service.full_refresh_every_seconds`, default 24h); failed full-refresh attempts no longer advance the success markers or anchor
- unattended provider fetch lanes now also honor health-check `refresh_full_snapshot` maintenance recommendations, so partial-coverage warnings from the latest health artifact can trigger the next fetch to self-upgrade into a conservative full refresh immediately instead of waiting for the cadence timer
- fresh unattended installs now also ship built-in mode-specific projected-request defaults for both Crunchyroll (`sync_fetch_crunchyroll`: `4` incremental / `55` full refresh) and HIDIVE (`sync_fetch_hidive`: `4` incremental / `71` full refresh) via the fetch-mode task projection table, so cold-start daemon budgeting does not assume either the ordinary fetch path or the heavier resweeps cost zero requests before history exists
- built-in projected-request defaults now consistently act as cold-start seeds rather than permanent hard overrides, so learned request history can take over for `mal_refresh`, `sync_apply`, and provider full-refresh fetches once the daemon has real observed cost data while explicit task-wide overrides still beat the shipped seeds
- overdue provider full-refresh requests no longer have to starve a fetch lane completely when the heavy sweep is temporarily over budget: the daemon now falls back to an incremental fetch when that cheaper mode fits, while leaving the full-refresh anchor/reason intact so the overdue resweep still happens later
- recommendation generation and metadata refresh exist
- discovery-candidate scoring now also records support-balance evidence (`best_single_source_votes`, `cross_seed_support_votes`, `effective_best_single_source_votes`, `effective_cross_seed_support_votes`, `support_balance_bonus`) so bursty one-seed recommendation spikes do not outrank steadier cross-seed consensus when aggregate support is otherwise similar; that support-spread bonus now also follows effective positive/current support after neutral/stale seed weighting instead of trusting raw vote spread alone, and discovery-target metadata refresh selection now follows that same balanced-support preference
- discovery-candidate ranking now also adds modest freshness and seed-activity weighting (`start_season`, `start_season_label`, `freshness_bucket`, `freshness_bonus`, `recent_seed_activity_bonus`, `freshest_supporting_seed_days`) so otherwise similar candidates lean toward newer catalog entries and recommendations backed by more recently active watch history instead of relying almost entirely on vote totals + overlap metadata
- materially old supporting seed activity now also gets a small explainable stale-support decay (`stale_supporting_seed_ids`, `stale_support_ratio`, `stale_support_penalty`) so stale taste evidence no longer looks like equally current consensus even when old seeds were once strongly rated
- stale-heavy multi-seed discovery support now also reduces the consensus-count boost itself via `base_effective_supporting_seed_count` and `stale_consensus_discount`, so older seed activity does not keep the same multi-seed confidence posture as equally broad fresh support
- discovery ranking now also carries a small explainable seed-quality calibration (`seed_quality_bonus`, `supporting_seed_scores`, `best_supporting_seed_score`) so support from highly scored MAL seeds or more deeply completed provider seeds can break near-ties without drowning out aggregate recommendation votes or broader cross-seed consensus
- discovery ranking now also carries a symmetric low-confidence seed penalty (`seed_quality_penalty`, `penalized_seed_scores`, `lowest_supporting_seed_score`) so candidates supported mainly by poorly scored/disliked seeds are tempered slightly instead of being treated as equally strong taste matches
- discovery recommendations now also suppress candidates backed only by explicitly dropped seeds or otherwise disliked-only support, while still allowing mixed-signal candidates to survive when genuinely stronger positive seed evidence exists
- mixed-signal discovery candidates now also discount dropped/disliked supporting seeds from their support-count boost and expose `negative_supporting_seed_ids`, `negative_support_ratio`, `effective_supporting_seed_count`, and `mixed_signal_penalty` so negative-heavy support lands below cleaner consensus instead of reading like equally strong multi-seed agreement
- discovery ranking now also treats explicitly neutral MAL seed scores conservatively: neutral-heavy support no longer contributes full multi-seed confidence, and recommendation metadata now exposes `neutral_supporting_seed_ids`, `neutral_support_ratio`, and `neutral_support_penalty` so neutral taste signals are visible instead of being silently treated like positive agreement
- discovery ranking now also applies a small explainable age decay to older MAL catalog entries (`freshness_penalty`, `catalog_age_in_seasons`) instead of only rewarding newer start seasons, so legacy titles lose tied/near-tied discovery races more honestly without becoming hard-filtered
- `list-mappings` now reflects the repo's multi-provider posture too: it lists all persisted mappings by default and supports explicit provider filtering instead of silently showing Crunchyroll only
- tests remain bundled in the repo for third-party auditing
