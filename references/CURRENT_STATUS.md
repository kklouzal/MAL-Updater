# Current Status

## Repository / packaging state

- The repo is treated as the canonical skill package (`SKILL.md` at repo root).
- Runtime state is externalized to `.MAL-Updater/` by default instead of living under the repo tree.
- The bootstrap/onboarding surface starts with `bootstrap-audit`, which now exposes provider readiness, per-provider operation-mode guidance/next-command hints, runtime-initialization readiness, user-daemon install/drift readiness, explicit manual-vs-daemon operation expectations, provider-intent/partial-bootstrap counts for partial-install edge cases, secrets-dir permission posture, blocking vs non-blocking onboarding counts, health-carried provider refresh plus mapping-review/backlog remediation, and explicit recommended commands for automation-friendly consumers.
- Repo-owned systemd automation is committed as a portable `mal-updater.service` template and rendered at install time by `scripts/install_user_systemd_units.sh`.
- The unattended model is now user-level systemd **daemon-first**, with internal Python control loops rather than timer-driven one-shot jobs.
- Public-repo hygiene is now part of the project contract: tracked code/references/tests/history should remain anonymized and free of personal identifiers, host-local absolute paths, and real secrets.

- recommendation generation and metadata refresh exist
- discovery-candidate scoring now also records support-balance evidence (`best_single_source_votes`, `cross_seed_support_votes`, `effective_best_single_source_votes`, `effective_cross_seed_support_votes`, `support_balance_bonus`) so bursty one-seed recommendation spikes do not outrank steadier cross-seed consensus when aggregate support is otherwise similar; that support-spread bonus now also follows effective positive/current support after neutral/stale seed weighting instead of trusting raw vote spread alone, and discovery-target metadata refresh selection now follows that same balanced-support preference
- discovery-candidate ranking now also adds modest freshness and seed-activity weighting (`start_season`, `start_season_label`, `freshness_bucket`, `freshness_bonus`, `recent_seed_activity_bonus`, `freshest_supporting_seed_days`) so otherwise similar candidates lean toward newer catalog entries and recommendations backed by more recently active watch history instead of relying almost entirely on vote totals + overlap metadata
- materially old supporting seed activity now also gets a small explainable stale-support decay (`stale_supporting_seed_ids`, `stale_support_ratio`, `stale_support_penalty`) so stale taste evidence no longer looks like equally current consensus even when old seeds were once strongly rated
- stale-heavy multi-seed discovery support now also reduces the consensus-count boost itself via `base_effective_supporting_seed_count` and `stale_consensus_discount`, so older seed activity does not keep the same multi-seed confidence posture as equally broad fresh support
- discovery ranking now also carries a small explainable seed-quality calibration (`seed_quality_bonus`, `supporting_seed_scores`, `best_supporting_seed_score`) so support from highly scored MAL seeds or more deeply completed provider seeds can break near-ties without drowning out aggregate recommendation votes or broader cross-seed consensus
- metadata-rich discovery near-ties now also get a small explicit affinity tie-break (`metadata_match_dimensions`, `metadata_affinity_bonus`) when cached MAL metadata lines up across multiple seed dimensions at once (genre/studio/source), and they now also carry bounded metadata-quality calibration (`metadata_quality_bonus`, `metadata_mean_band`, `metadata_popularity_band`) so stronger MAL mean / broader adoption can break otherwise flat metadata-rich ties without replacing the core vote/support model
- discovery candidates now also expose a small global catalog quality/adoption calibration (`catalog_quality_bonus`, `catalog_mean_band`, `catalog_popularity_band`) even outside metadata-rich ties, so flatter recommendation races can still break a little more honestly on elite MAL mean or broader adoption without pretending those catalog signals outweigh consensus support
- discovery ranking now also carries a symmetric low-confidence seed penalty (`seed_quality_penalty`, `penalized_seed_scores`, `lowest_supporting_seed_score`) so candidates supported mainly by poorly scored/disliked seeds are tempered slightly instead of being treated as equally strong taste matches
- discovery recommendations now also suppress candidates backed only by explicitly dropped seeds or otherwise disliked-only support, while still allowing mixed-signal candidates to survive when genuinely stronger positive seed evidence exists
- mixed-signal discovery candidates now also discount dropped/disliked supporting seeds from their support-count boost and expose `negative_supporting_seed_ids`, `negative_support_ratio`, `effective_supporting_seed_count`, and `mixed_signal_penalty` so negative-heavy support lands below cleaner consensus instead of reading like equally strong multi-seed agreement
- discovery ranking now also treats explicitly neutral MAL seed scores conservatively: neutral-heavy support no longer contributes full multi-seed confidence, and recommendation metadata now exposes `neutral_supporting_seed_ids`, `neutral_support_ratio`, and `neutral_support_penalty` so neutral taste signals are visible instead of being silently treated like positive agreement
- discovery ranking now also applies a small explainable age decay to older MAL catalog entries (`freshness_penalty`, `catalog_age_in_seasons`) instead of only rewarding newer start seasons, so legacy titles lose tied/near-tied discovery races more honestly without becoming hard-filtered
- `list-mappings` now reflects the repo's multi-provider posture too: it lists all persisted mappings by default and supports explicit provider filtering instead of silently showing Crunchyroll only
- tests remain bundled in the repo for third-party auditing
